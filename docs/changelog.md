### Removed
- **Sprint-promotion automation (PROC-002 + PROC-003, reverted)**: Deleted `scripts/promote-sprint-item.mjs` and `scripts/promote-sprint-item.test.mjs` — the regex-based transforms that tried to automate the NEXT.md / ROADMAP.md / changelog hand-off after a PR ships. In practice the promotions had too many edge cases the transforms couldn't cover cleanly: (1) bundled PRs leaked `(bundled)` suffixes into ledger reads; (2) queue slots are the script's source for scope prose, but most queue items live as one-line entries in `NEXT.md` and a full spec in `ROADMAP.md` — the script couldn't fetch the latter without bespoke per-id logic; (3) the Recently-completed row cap and per-row title fields drifted from the queue's title format, so manual cleanup was always needed after every run. The canonical sprint hand-off is now the expanded manual checklist in `REVIEW.md § Sprint Tracker Hand-off` (PR #10). The original `### Added` entries for PROC-002 (PR #8) and PROC-003 (PR #9) below are preserved for audit continuity — treat both as superseded by this entry.

### Added
- **Confidence-based auto-approval (AUTO-003b)**: New auto-approval pipeline for AI-generated tests. Projects gain an optional `autoApproveThreshold` (0–1, validated on `PATCH /api/v1/projects/:id`) — when set, tests whose `confidenceScore` (sourced from the deduplicator's existing `_quality` score on `backend/src/pipeline/deduplicator.js`) meets or exceeds the threshold are auto-approved during persistence (`backend/src/pipeline/testPersistence.js`) instead of landing in Draft. Migration `017_auto_approval.sql` adds five provenance columns to `tests` (`confidenceScore`, `approvalSource`, `approvalThreshold`, `approvedAt`, `approvedBy`) plus `autoApproveThreshold` on `projects`; `approvedAt` is epoch ms (per the migration's INTEGER type) so the approvals timeline can do straight arithmetic ranges, while `reviewedAt` stays ISO-string to match the rest of the codebase. Auto-approval emits a `test.auto_approve` activity-log entry (imperative form, matching the existing `test.create` / `test.approve` / `test.reject` / `test.restore` / `test.delete` / `test.generate` convention) for the audit trail. New `POST /api/v1/tests/:testId/revoke` (`qa_lead`+, registered in `backend/src/middleware/permissions.json`) returns an approved test (auto- or human-approved) to draft, clearing all six review/provenance columns and logging a `test.revoke` activity. New `GET /api/v1/projects/:id/approval-stats` returns `{ human, auto, draft, total, revertRate7d, autoApprovals7d, reverts7d }` counts for the project-settings calibration UI. Frontend ships `api.revokeApproval(testId)` and `api.getApprovalStats(projectId)` helpers in `frontend/src/api.js`; the Test Detail sidebar's "Move to Draft" button is replaced with a "Revoke approval" button (`frontend/src/pages/TestDetail.jsx`) that calls the new endpoint and surfaces the auto-approval threshold in its tooltip; the Tests list (`frontend/src/pages/Tests.jsx`) renders a two-tone badge — green "Approved" plus a 🤖 sub-badge — when `approvalSource === "auto"`. **Ops kill-switch**: the `DISABLE_AUTO_APPROVAL` env var (any truthy value — `1` / `true` / `yes`, case-insensitive) forces every generated test to land in Draft regardless of per-project thresholds, so ops can roll back the feature on a bad AI-provider day with a restart instead of a deploy; per-project thresholds stay intact and re-activate when the env var is cleared (documented in `backend/.env.example` and `docs/guide/auto-approval.md`). **Sidebar "🤖 N auto today" badge** on the `/approvals` nav entry (`frontend/src/components/layout/Sidebar.jsx`) counts today's auto-approvals (since local-midnight) and matches the rejection-criterion in NEXT.md:100 — rail-mode shows a corner dot + tooltip, expanded mode shows the count pill. **Shared TanStack Query hook** `frontend/src/hooks/queries/useAutoApprovalsQuery.js` deduplicates fetches between the sidebar badge and the ReviewQueue's 24h auto-approvals tray: mounting both surfaces is still one in-flight request, and mutations that affect the feed (`TestDetail.handleRevokeClick`, `ApprovalsTimeline.handleRevoke`, `ReviewQueue` bulk restore) call `invalidateAutoApprovalsCache()` to refresh both at once. **Service layer** `backend/src/services/approvalService.js` exports `PROVENANCE_CLEAR` (frozen shape used by single restore, bulk restore, and revoke — so all three "return to draft" paths stay byte-identical), `humanApproval(actorInfo)` (for manual approvals), `APPROVAL_SOURCE` enum, and `computeStats(projectId)` (SQL-aggregated status counts + 7-day revert rate via two `COUNT(DISTINCT testId)` queries on the activity log, replacing a ~20 MB in-memory scan). **Shared cross-side module** `shared/activityTypes.js` at the repo root exports `ACTIVITY_TYPES` — both backend and frontend import from one file so a typo becomes a ReferenceError rather than a silent empty-feed regression (the prior `"test.approve"` vs `"test.approved"` bug was impossible after this). Vite serves it via `server.fs.allow` in `frontend/vite.config.js`. New unit test `backend/tests/auto-approval.test.js` + HTTP integration `backend/tests/auto-approval-routes.test.js` (both registered in `backend/tests/run-tests.js`) cover null-threshold, below-threshold, above-threshold (with activity-log assertion), revoke-auto-approved, revoke-human-approved, cross-workspace ACL, idempotency guard (drafts can't be revoked), and concurrent-revoke race (two simultaneous requests serialise to exactly one 200 + one 400, with one audit row). Frontend API contract `frontend/tests/approval-provenance.test.js` locks the `revokeApproval` + `getApprovalStats` request/response shape. User-facing docs at `docs/guide/auto-approval.md` cover how to tune the threshold, revoke paths, the kill-switch, permissions, and troubleshooting; sidebar-linked in `docs/.vitepress/config.mjs`.
- **Sprint promotion auto-prune (PROC-003)**: `scripts/promote-sprint-item.mjs` (PROC-002) is extended with three new transforms that fold the most-frequently-missed manual steps from REVIEW.md § Sprint Tracker Hand-off into the script. (1) `appendCompletedWorkSummary()` adds a row to `ROADMAP.md`'s `## Completed Work Summary` table for each shipped id (bundled IDs like `AUTO-017.3 + PROC-001 + PROC-003` split into one row per id). (2) `decrementRemainingCounts()` finds the shipped item's category in the bottom-of-file `## Summary` stats table (via prefix-based inference: `AUTO-*` → Autonomous Intelligence, `PROC-*` → Process automation, etc.), bumps `✅ Done` and decrements `🔲 Pending`, then recomputes the `**Totals**` row from the resulting per-category numbers. (3) `pruneShippedRoadmapEntry()` deletes the shipped item's detailed `### <ID> — <title>` section so the Completed Work Summary table is the canonical record. All three transforms are best-effort and idempotent — re-running the script never double-writes the table row, never doubles the decrement, and skips items whose section/heading isn't found rather than crashing. Orphan ID prefixes (`MET-*`, `UI-REFACTOR-*`) emit a `console.warn` and skip the Summary decrement. Sandbox test in `scripts/promote-sprint-item.test.mjs` is extended with four new assertions covering the appended row, decremented counts (Autonomous Intelligence row + recomputed Totals), the pruned-detailed-entry survival of a sibling item, and the idempotency case (running twice doesn't double-write).
